# Patent Technology Roadmap & Landscape Analyzer

四阶段全流程分析：数据准备与标引 → 价值评估 → 竞争态势与布局分析
→ 解读与汇报。

## 输入格式

本 skill 接受两种输入：

**A) patent-classifier 丰富版 Excel 输出（推荐,快速路径）**

识别标志：Excel 中包含"主标签"+"置信度"+"分类来源"列。

→ 跳过第一阶段的数据采集和分类步骤
→ 直接消费 classifier 输出的主标签、次要标签、置信度、同族组、有效日期等
→ 仅需执行 1.1 明确分析范围 + 1.3 数据清洗（精简版），然后进入第二阶段

**B) 原始专利清单（标准路径）**

Excel/CSV/内联文本，含专利号 + 可选的标题、摘要、申请日等。

→ 内部调用 classifier 完成数据采集和分类
→ 完整执行第一阶段后进入第二阶段

## Workflow Overview

```
Input (patent list)
  → 第一阶段: 数据准备与标引
  │   1.1 明确分析范围
  │   1.2 数据采集与分类（委托 classifier）
  │   1.3 数据清洗（利用同族组增强去重）
  │
  → 第二阶段: 价值评估
  │   2.1 综合价值分 (同族50% + 被引20% + 法律状态20% + 异议10%)
  │   2.2 价值等级分层
  │
  → 第三阶段: 竞争态势与布局分析
  │   3.1 六大区域检测 (红海/壁垒/优势/空白/断层/转移)
  │   3.2 产品关联标注
  │   3.3 技术问题竞争矩阵 (Problem × Assignee × Approach)
  │   3.4 Claim 覆盖范围分析 (高价值专利深度比对)
  │
  → 第四阶段: 解读与汇报
      4.1 技术路线演进分析（问题域聚类 → 时间线路线图）
      4.2 近18个月新申请专利解读与下一代技术启示
      4.3 专利布局建议清单 (Filing Opportunity Sheet)
      4.4 辅助清单（高价值 & 风险专利）
      4.5 Excel 输出 + PPT 汇报
```

# 第一阶段: 数据准备与标引

此阶段是整个分析的基础。通过委托 classifier 执行分类标引，大幅减少
重复工作。

## 1.1 明确分析范围

在开始分析前，确认以下范围参数（如用户未明确，主动询问或从数据推断）：

- **技术领域**: 从用户提供的专利清单推断，或由用户明确指定技术分支
- **时间范围**: 通常近 5-10 年，结合产品生命周期调整
- **竞争对手**: 从专利清单的申请人字段提取，建议聚焦不超过 3 家主要友商
- **地域**: 从专利号的国别前缀（US/CN/EP/JP/KR）推断布局重点

## 1.2 数据采集与分类

**本步骤根据输入类型走不同路径：**

### 路径 A — Classifier 丰富版输出作为输入

如果输入 Excel 包含"主标签"+"置信度"+"分类来源"列（classifier 输出的标志列），
则所有元数据和分类结果已可用：

```
可直接消费的 classifier 输出列:
  - 主标签 → roadmap 的分类依据，驱动按类别聚类
  - 次要标签 → 4.1 跨域关联分析的结构化输入
  - 置信度 → 低置信度专利在聚类时标注"分类待确认"
  - 同族组 → 1.3 去重步骤的辅助依据
  - 有效日期 → 时间线分析
  - 标题/摘要/申请人/CPC → 元数据复用
  - 分类来源 → 报告标注
```

**直接跳过数据采集和分类**，进入 1.3 数据清洗。

### 路径 B — 原始专利清单，用户已有标题+摘要

调用 classifier，classifier 会在 Step 0.3 检测到"数据充分"，
跳过搜索，直接执行分类标引。

**搜索消耗 ≈ 0**，仅消耗分类推理的 token。

### 路径 C — 原始专利清单，缺少标题/摘要

调用 classifier 完整流程（Step 0-5），classifier 会执行数据检索 +
分类标引，输出丰富版结果。

**搜索消耗**: min(需检索专利数 × 2, 30) 次

### 三种路径共同的后续处理

无论哪种路径，分类完成后都需要：

1. **确认分类结果**：向用户展示分类汇总（主标签分布、置信度分布），
   确认后再进入下一阶段

2. **补充 roadmap 特有字段**：如果用户数据中包含以下字段但 classifier
   输出中没有，需要保留并合并：
   - 技术问题 / 技术手段 / 技术功效（用户已提供则直接使用，不重新推导）
   - 引证信息、法律状态、同族数量（roadmap 评分需要）

3. **确定有效日期**：
   有效日期 = 最早优先权日（如有）> 申请日 > 公开日。
   Classifier 输出中已有此字段，直接使用。

**推荐导出字段**（用户自行采集时的建议）：公开号、申请日、优先权日、标题、
摘要、权利要求、申请人、发明人、同族信息、引证信息、法律状态、**技术标签**
（如已有分类结果）。

## 1.3 数据清洗

在分析前执行以下清洗步骤：

### 1.3.1 去重

同一专利家族仅保留一件代表（通常为最早优先权或最重要同族）。

**利用 classifier 输出的同族组字段增强去重：**

```
去重策略:
  1. 优先使用 classifier 输出的"同族组"字段
     → 同组内保留最早优先权/最重要同族作为代表
  2. 如果同族组字段为空，退回传统判断:
     → 相同优先权日 + 相同申请人 + 近似标题 → 视为同族
  3. 如果 roadmap 在后续分析中发现 classifier 未识别的同族关系
     （如引证关系、连续申请号），补充标注
```

### 1.3.2 申请人标准化

统一名称变体。如"华为技术有限公司"="HUAWEI"="Huawei Technologies"；
"NVIDIA Corp."="NVIDIA Corporation"。为每个申请人建立规范名。

### 1.3.3 排除无关

剔除外观设计专利、已放弃且无延续的专利、明显超出分析技术领域的专利。

# 第二阶段: 价值评估

**详见 `references/quality-scoring.md`。**

为每件专利计算 0-100 的综合价值分，采用四维加权评分：

| 指标 | 权重 | 说明 |
|------|------|------|
| 同族数量 | 50% | ≥5国=100, 3-4国=70, 2国=40, 1国=20 |
| 被引次数 | 20% | ≥20=100, 10-19=80, 5-9=60, 1-4=30, 0=10 |
| 法律状态 | 20% | 授权维持=100, 实审中=60, 公开=40, 驳回/放弃=0 |
| 经历异议/无效 | 10% | 经挑战维持=100, 未经挑战=60, 被无效=0 |

```
Score = 0.50×Family + 0.20×Citation + 0.20×Legal + 0.10×Challenge
```

### 数据不完整时的处理

- 从专利号推断法律状态：US...B2=已授权，CN...A=公开，US...A1=申请公开
- 同族数量：清单中同一申请人有 US+CN 同标题专利 → 至少 2 个同族；
  也可参考 classifier 输出的同族组（同组专利数 ≈ 最小同族数）
- 引证和异议数据缺失 → 按中间分计（引证=30，异议=60）
- 完全无法判断 → 默认 50 分，标注"数据不足"

### 价值等级

| 等级 | 分数 | 标签 | 气泡大小 |
|------|------|------|----------|
| ★★★ | 75-100 | 核心专利 | 大 |
| ★★ | 50-74 | 重要专利 | 中 |
| ★ | 25-49 | 外围专利 | 小 |
| — | 0-24 | 低价值 | 点 |

# 第三阶段: 竞争态势与布局分析

## 3.1 六大战略区域检测

**详见 `references/strategic-zones.md`。**

在专利数据上识别并标注六种典型区域：

| 区域 | 特征 | 战略含义 |
|------|------|----------|
| 🔴 红海激战区 | ≥2 方气泡密集重叠，近 2-3 年集中 | 研发焦点，侵权高风险，需规避或无效 |
| 🔵 友商壁垒区 | 某方气泡绝对主导且气泡大 | 对手核心领地，慎入，考虑许可或规避 |
| 🟢 我方优势区 | 我方气泡密集，友商稀少 | 主动维权或交叉许可筹码 |
| ⬜ 战略空白区 | 无/零星小气泡，但有商业前景 | 研发投入机会，抢先布局 |
| 🟡 技术断层区 | 早期有气泡，近 2-3 年消失 | 技术淘汰或瓶颈，需判断是否跟进 |
| ➡️ 技术转移 | 近两年气泡从 A 分支向 B 分支移动 | 对手研发方向变化，提前应对 |

## 3.2 产品关联标注

同步执行产品关联分析：

对每件专利，基于申请人搜索是否有对应产品/特性：
- 🔗 已落地: 找到具体产品+特性匹配技术方案
- 🔍 疑似相关: 同领域产品但无法确认具体技术
- ➖ 未发现: 公开信息无匹配

搜索策略：按"申请人 × 问题域"分组搜 2-4 次，匹配技术机制而非仅领域；
注意时间对齐和产品世代。

**次要标签辅助**：有次要标签的专利，产品关联搜索可扩展到次要领域对应的产品线。

### 单一申请人模式

全部专利属同一申请人时：跳过区域 1-3（无竞争对比），聚焦区域 4-6
及"产品覆盖缺口"（有专利无产品关联 → 可能是前沿预研或已放弃方向）。

## 3.3 技术问题竞争矩阵 (Problem-Competitor Matrix)

战略区域识别回答的是"哪个技术分支竞争激烈"——粒度太粗。竞争矩阵
下沉到**技术问题**层面，回答"在同一个问题上，各家走了什么路、谁还没进场"。

> 这一步是从"数据对比"到"布局启示"的关键桥梁。

### 3.3.1 构建矩阵

前置条件：第四阶段 4.1 的"按问题域聚类"已完成（可提前执行 4.1 的聚类
子步骤，再回到 3.3 构建矩阵）。

对每个主标签下的每个问题域，构建如下矩阵：

```
主标签: L3 互连协议
问题域: 链路训练中的时序收敛问题

           | 我方        | NVIDIA       | Intel        | AMD
-----------+-------------+--------------+--------------+------------
专利数量    | 2           | 4            | 1            | 0  ←空白
技术路径    | 自适应EQ     | 自适应EQ+ML  | 静态预设      | —
代表专利    | CN...B      | US...B2      | US...B2      | —
方案代际    | 第2代        | 第3代(最新)   | 第1代         | —
最新申请年  | 2022        | 2024         | 2021         | —
价值分均值  | 65          | 82           | 45           | —
```

**置信度加权**：低置信度专利在矩阵计数中给 0.5 权重（而非 1.0），
用括号标注如 "(1)"，避免分类不确定的专利扭曲竞争态势判断。

### 3.3.2 标注技术路径 (Approach)

**技术路径**是竞争矩阵的核心——不是简单统计"谁有几件专利"，而是提炼
各申请人的**解题思路差异**。

提取方法：
1. 从聚类内各专利的"技术手段"字段提炼共性方案（用户已提供则直接用）
2. 按方案类型归纳：硬件方案 / 软件方案 / 算法方案 / 混合方案 / 
   材料方案 / 架构方案 等
3. 同一申请人在同一问题域内可能有多条路径（说明其多路押注）
4. 标注方案代际：第1代(基础方案) / 第2代(改进) / 第3代(最新演进)

**代际判断依据**：
- 时间线先后（越晚≠越新，但通常相关）
- 方案复杂度递增或范式转换
- 后代专利引用前代专利
- 产品侧验证（产品采用的是哪代方案）

### 3.3.3 识别四类竞争态势

从矩阵中识别以下四种关键态势（比六大战略区域更精确）：

**① 路径空白 (Approach Gap)**
- 某个问题域中存在已被验证的技术路径，但特定申请人未覆盖
- 例：NVIDIA 在"CXL一致性"上只有硬件仲裁方案，无协议层方案
- 布局价值：中高（可参考已有路径快速布局）

**② 路径真空 (Approach Vacuum)**
- 某个问题域中，所有申请人都未探索的技术路径
- 例：所有人走硬件方案，无人尝试软硬协同方案
- 布局价值：最高（但需验证技术可行性）

**③ 代际差距 (Generation Gap)**
- 同一路径上，友商已演进到第3代，我方停留在第1代
- 例：NVIDIA 的链路训练已引入ML辅助（第3代），我方仍用传统自适应EQ
- 风险级别：高（我方专利可能被友商新方案规避）

**④ 收敛趋势 (Convergence Signal)**
- 多个申请人的最新专利从不同路径趋向相似的技术方案
- 例：各家的最新专利都开始采用"硬件加速+软件策略"混合架构
- 布局价值：中（说明行业共识形成，需要布局该方向的改进方案）

### 3.3.4 单一申请人模式

只有一个申请人时，矩阵退化为"问题域 × 技术路径"二维表：
- 识别该申请人在各问题域上的路径覆盖完整度
- 标注哪些问题域只有单一路径（风险：被规避）
- 标注哪些问题域有多路径覆盖（优势：形成专利网）
- 与产品关联交叉：有路径但无产品 → 前沿预研 or 已放弃

## 3.4 Claim 覆盖范围分析 (Claim Coverage Mapping)

**详见 `references/claim-coverage.md`。**

竞争矩阵基于摘要和技术手段分析，能看出"各家解决了什么问题、走了什么
路径"。但专利的实际保护范围取决于**权利要求**（Claim），而非摘要。

此步骤对高价值专利做更深一层的 Claim 级比对，发现**Claim Gap**——
所有竞争者的权利要求都没有覆盖到的技术方案变体。

### 3.4.1 选择分析对象

Claim 分析成本高（每件需要细读 Claim 1 并分解特征），仅对以下专利执行：

- 🔴 红海激战区内，综合价值分 ≥ 65 的专利（★★★ 和高 ★★）
- 🔵 友商壁垒区内，综合价值分 ≥ 75 的友商核心专利
- 用户明确指定需要深入分析的专利

**数量控制**: 每个问题域选取不超过 3-5 件核心专利做 Claim 分析。
全局上限 15 件。超出时优先覆盖红海区。

### 3.4.2 Claim 1 特征分解

复用 **seek-cfp-patent** skill 的分解方法论：

1. 提取 Claim 1 全文（从用户数据或 web search 获取）
2. 分解为 F0 [前序/限定] + F1..Fn-1 [核心特征] + Fn [限定特征]
3. 对每个特征，标注：
   - 特征类型：结构特征 / 方法步骤 / 功能限定 / 参数限定
   - 保护宽度：宽（功能性描述）/ 中（方法步骤）/ 窄（具体参数/结构）

### 3.4.3 同问题域内 Claim 横向比对

对同一问题域内选取的多件核心专利，进行 Claim 特征横向对齐：

```
问题域: 链路训练中的时序收敛

           | NVIDIA US...B2     | Intel US...B2      | 我方 CN...B
-----------+--------------------+--------------------+------------------
F0 前序    | 链路训练方法        | 信号均衡方法        | 自适应训练方法
F1 核心1   | 多阶段自适应EQ      | 固定预设系数查表     | 双环路自适应EQ
F2 核心2   | ML模型预测最优参数   | —                  | —
F3 核心3   | 实时反馈调节        | —                  | 收敛检测机制
Fn 限定    | 特定信号速率≥32GT/s | PCIe Gen5          | PCIe Gen4/5
-----------+--------------------+--------------------+------------------
保护宽度    | 宽(功能性+ML)      | 窄(具体参数)        | 中(方法步骤)
```

### 3.4.4 识别 Claim Gap

从横向比对中识别以下三类 Claim 层面的布局机会：

**① 特征空白 (Feature Gap)**
- 某个核心技术特征维度，所有已有 Claim 都未涉及
- 布局方向：围绕该特征维度撰写独立权利要求

**② 参数区间空白 (Parameter Gap)**
- 已有 Claim 通过具体参数限定了保护范围，但存在参数空白区间
- 布局方向：覆盖不同参数区间的方案实现

**③ 实现层级空白 (Abstraction Gap)**
- 已有 Claim 集中在某个实现层级，其他层级未被覆盖
- 布局方向：在不同抽象层级撰写对应的方法/装置权利要求

### 3.4.5 Claim 分析结果输出

将 Claim Gap 发现合并输入 4.3 布局建议清单，标注来源为"Claim级分析"。
在 Excel 中，Claim 分析结果输出到 **Sheet 11: Claim 覆盖分析**。

### 3.4.6 搜索预算

- 获取 Claim 1 全文：每件专利 1-2 次搜索（Google Patents / 佰腾）
- 总额外预算：min(分析专利数 × 2, 20)
- 如用户已提供权利要求数据 → 搜索量为零

# 第四阶段: 解读与汇报

## 4.1 技术路线演进分析

对每个主标签类别，执行以下子步骤：

### 提取技术三要素

每件专利提炼：**技术问题**（一句话）、**技术手段**（核心方案 1-2 句）、
**技术功效**（声称优势）。用户已提供则直接使用。

### 按问题域聚类

同一主标签内按具体子问题聚类：
- 相近问题陈述合并
- 命名用"问题导向"标签（如"PCIe链路训练中的时序收敛问题"而非"PCIe相关"）
- 粒度：5-15 件专利→2-5 个聚类

**低置信度专利处理**：置信度为"低"的专利在聚类时附注"（分类待确认）"，
聚类后如发现其与聚类主题明显不符，可重新审视其分类。

### 构建时间线路线图

每个问题域内按有效日期排列，识别技术演进阶段：

```
主标签: {类别}  |  问题域: {名称}
核心问题: {描述}
时间跨度: {最早} → {最新}

阶段 1: {标签} ({时间段})
  - {专利号} ({日期}) [★★★ 85分]: {技术手段} → {功效}
    🔗 关联产品: {产品} — {特性}
  技术路线特征: {本阶段特征}

阶段 2: {标签} ({时间段})
  - {专利号} ({日期}) [★★ 62分]: {技术手段} → {功效}
  技术路线特征: {与上一阶段的差异}
```

阶段不强制划分——有明显转折点时才分阶段。

### 跨域关联

所有类别完成后，识别跨域关联：

1. **类内聚类间关联**：同一主标签下不同问题域的技术互补
2. **跨主标签依赖**：不同主标签间的技术依赖
3. **技术融合信号**：不同问题域趋于收敛

**利用 classifier 输出的次要标签增强跨域关联识别：**

```
跨域关联识别:
  扫描所有有次要标签的专利。
  如果主标签归 L3、次要标签归 L4 → 标注 L3↔L4 跨域关联。
  如果多件专利共享同一跨域模式 → 识别为"技术融合信号"。
  相比从 rationale 文本中挖掘，次要标签提供结构化、可靠的跨域信号。
```

### 趋势分析

1. **技术演进趋势**: 方案变化方向（复杂化/简化、硬件↔软件、集成↔专精）
2. **申请密度趋势**: 加速=活跃研发，减速=成熟或转向
3. **申请人变化**: 同一实体持续申请 vs 新玩家进入
4. **产品落地节奏**: 专利到产品的时间差，最新专利是否领先于产品
5. **技术融合信号**: 不同问题域是否趋于收敛

## 4.3 专利布局建议清单 (Filing Opportunity Sheet)

这是整个分析最终的"so what"——将竞争矩阵和近期专利解读的发现转化为可执行的布局行动。

> 原则：每条布局建议必须指向一个**具体的技术问题 + 具体的技术路径方向**，
> 而非笼统的"加强XX领域研发投入"。

### 4.3.1 布局机会识别

从 3.3 竞争矩阵的四类态势中提取布局机会：

| 来源态势 | 布局机会类型 | 优先级权重 |
|----------|------------|-----------|
| ① 路径空白 | 快速跟随布局：参考已有路径，在我方未覆盖的技术方案上申请 | 中高 |
| ② 路径真空 | 先发创新布局：所有人都没走过的路径，抢先占位 | 最高（需验证可行性） |
| ③ 代际差距 | 追赶式布局：沿友商已演进的路径方向，布局更新一代方案 | 高 |
| ④ 收敛趋势 | 卡位式布局：在行业共识方向上布局改进方案和变体 | 中 |

此外，从其他分析步骤补充：
- ⬜ 战略空白区 → 探索性布局机会
- 🟢 我方优势区的边缘 → 扩展性布局（巩固护城河）
- 🔮 4.2 近期专利解读的布局启示 → 前瞻性布局（对标友商最新方向或抢占技术拐点）

### 4.3.2 优先级评分

每条布局机会按三个维度打分（各 1-5 分），计算综合优先级：

```
布局优先级 = 商业价值(40%) × 竞争空白度(35%) × 技术可行性(25%)
```

**商业价值** (1-5):
- 5: 问题域直接关联当前主力产品的核心特性
- 4: 关联下一代产品的已知规划方向
- 3: 关联产品路线图中的潜在方向
- 2: 技术有前景但产品路线尚不明确
- 1: 纯前沿探索，短期无产品关联

**竞争空白度** (1-5):
- 5: 路径真空（无人涉足的方案方向）
- 4: 路径空白（我方空白但有可参考路径）
- 3: 代际差距（友商有布局但我方可追赶）
- 2: 收敛趋势（多方趋同，但仍有改进空间）
- 1: 红海方向（差异化空间极小）

**技术可行性** (1-5):
- 5: 我方已有相关技术储备和产品实践
- 4: 技术储备部分匹配，需要适度研发
- 3: 需要新研发但技术路径清晰
- 2: 技术路径需要探索，有不确定性
- 1: 高度实验性，技术风险大

### 4.3.3 建议 Claim 核心要素

对优先级 P0/P1 的布局机会，进一步给出建议的权利要求核心要素方向：

```
布局方向: CXL 设备中结合硬件状态机与软件策略的一致性管理
建议 Claim 核心要素:
  - 前序: 一种用于 CXL 互连的内存一致性管理方法/装置
  - 核心特征方向: 硬件状态机处理热路径一致性请求 + 软件策略引擎
    处理冷路径/异常路径
  - 差异化要素: 相对于纯硬件方案(NVIDIA CN...B)的优势在于可编程
    灵活性；相对于纯协议方案(Intel US...B2)的优势在于热路径延迟
  - 需规避的已有专利: {列出可能冲突的友商专利号}
```

注意：这里给出的是**方向性建议**，不是可直接提交的权利要求文本。
最终 Claim 撰写需要发明人确认技术细节 + 代理人规范化语言。

### 4.3.4 输出格式

布局建议按优先级排序，分三个梯队：

```
P0 — 立即启动 (综合分 ≥ 4.0):
  应在下一个季度的发明挖掘/交底书撰写中优先安排

P1 — 计划布局 (综合分 3.0-3.9):
  纳入半年度专利布局规划，匹配研发进度后安排

P2 — 观察储备 (综合分 < 3.0):
  持续跟踪该方向的行业动态，待条件成熟后升级
```

## 搜索预算全局规划

Roadmap 的搜索消耗分为三个阶段：

```
阶段 1 — Classifier 分类（仅标准路径 B/C 需要）:
  路径 B（用户有标题+摘要）: ≈ 0 次
  路径 C（需检索）: min(需检索数 × 2, 30) 次

阶段 2 — 产品关联 (3.2):
  按"申请人 × 问题域"分组搜 2-4 次

阶段 3 — 近期专利解读 (4.2):
  公开宣传记录 + 技术信号验证: min(入选专利数 × 2, 20) 次

阶段 4 — Claim 分析 (3.4, 可选):
  min(分析专利数 × 2, 20) 次

总预算上限: ≤ 80 次
```

数据越丰富 → 阶段 1 消耗越少 → 更多预算留给深度分析。

## Practical tips

1. **快速路径优先**: 如果用户已有 classifier 丰富版输出，直接用作输入，
   可跳过数据采集和分类阶段。这是最高效的工作流。

2. **用户数据优先**: 有技术问题/手段/功效字段则直接使用，不重新推导。

3. **日期优先级**: 优先权日 > 申请日 > 公开日。Classifier 输出的
   有效日期字段已按此优先级处理。

4. **预分类加速聚类**: 先分类再聚类，在已知技术域内找子问题更精准。

5. **大批量(>30件)**: 先完成全部标引，再逐类别做路线分析。
   超大量建议分批 15-20 件。

6. **评分数据不足**: 默认 50 分并标注，不跳过评分步骤。

7. **单一申请人**: 跳过竞争态势区域(1-3)和矩阵的竞争列，聚焦路径覆盖
   完整度和产品覆盖缺口。布局建议聚焦"路径真空"和"扩展性布局"。

8. **语言**: 匹配用户语言，中文专利组合默认中文输出。

9. **分类校验**: 利用 classifier 输出的置信度字段快速定位需要
   人工复核的分类结果，比逐件扫描更高效。

10. **竞争矩阵执行顺序**: 3.3 依赖 4.1 的问题域聚类结果。实际执行时
    先跑 4.1 聚类 → 回到 3.3 构建矩阵 → 再完成 4.1 剩余的趋势分析。

11. **布局建议的技术可行性**: 如无法判断我方技术储备，默认 3 分并标注
    "需研发团队确认"。不因信息不足而跳过布局建议。

12. **Claim 核心要素的边界**: 给出的是方向性建议，不是法律文本。
    明确标注"最终需发明人确认技术细节 + 代理人规范化"。

13. **Claim 分析的选择性执行**: 3.4 是可选深度分析，仅对红海区/壁垒区的
    高价值专利执行。如用户未提供权利要求数据且专利数量 >20，建议仅对
    Top 5 核心专利做 Claim 分析。

14. **近期专利解读的时间窗口**: 18个月是默认值，可根据技术领域的
    专利申请周期调整（快速迭代领域可缩短至12个月，基础研究可延长至24个月）。

15. **近期专利解读是布局建议的重要输入**: 4.2 的"布局启示"应合并到
    4.3 布局建议清单中，标注来源为"近期专利解读"。

16. **次要标签是跨域分析的加速器**: 比从 rationale 文本中挖掘更可靠，
    4.1 跨域关联识别应优先利用。
