# Patent Monitor — 竞争对手新公开专利定期监控分析

对一批时间范围内友商新公开的专利进行系统性分析：按预设重要技术领域分类、逐篇
评估重要性、关联友商产品线、识别"预设领域外但值得关注"的专利，最终输出结构化
Excel 报告。

## 定位与互补关系

本 skill 专注**定期批量监控**，与其他专利 skill 互补：
- **patent-classifier**: 本 skill 调用其完成领域分类和领域预筛
- **seek-cfp-patent**: 本 skill 发现的高重要性专利可输出给 CFP skill 做深入侵权分析
- **patent-roadmap**: 本 skill 的分领域清单可输入 roadmap skill 做技术演进分析

---

## 输入格式

本 skill 接受两种输入：

**A) patent-classifier 丰富版 Excel 输出（推荐，快速路径）**

识别标志：Excel 中包含"主标签"+"置信度"+"分类来源"列。

→ 跳过 Step 1 的分类环节，直接消费 classifier 输出
→ classifier 输出中的 `领域范围` 列提供 In-Scope/Out-of-Scope 判定
→ 进入 Step 2 重要性评分

**B) 原始专利清单（标准路径）**

`.xlsx` / `.csv` / 内联文本，预期为已从专利数据库导出的丰富数据，
典型字段包括：

| 字段 | 必需 | 说明 |
|------|------|------|
| 专利号 | ✓ | 如 US12334926B2, CN116366486A |
| 标题 | ✓ | 专利标题（中文或英文） |
| 摘要 | ✓ | 专利摘要 |
| 申请人 | ✓ | 专利权人名称 |
| 申请日/公开日 | 推荐 | 用于时间分布分析 |
| 同族数量 | 推荐 | 简单同族或 INPADOC 同族大小 |
| 布局国家 | 推荐 | 如 "US, CN, EP, JP, KR" |
| 法律状态 | 可选 | 有效/审查中/授权等 |
| IPC/CPC | 可选 | 可辅助分类 |
| 第一权利要求 | 可选 | 有则可做更精确的技术分析 |

如果输入缺少"标题"或"摘要"，提示用户补充——这两个字段是分类和产品关联的基础。
如果"同族数量"或"布局国家"缺失，在重要性评分中跳过对应维度并告知用户。

→ 内部调用 classifier 完成列名映射、领域预筛、分类标引
→ 然后进入 Step 2

### 可选输入

- **重点关注领域**：用户可指定本期重点关注的技术领域子集（如"本季度重点看 L3
  互联协议 和 L10 安全"）。未指定则使用完整 L0-L10 体系。
- **目标友商列表**：默认 NVIDIA、Intel、AMD、海光、寒武纪。用户可增删。
- **分析时间范围**：如 "2025 Q1"，用于报告标题和筛选。

## Workflow

```
Step 0   输入解析与路径判断
Step 1   领域预筛 + 技术分类（委托 classifier）
Step 2a  三维暂估评分（同族+布局+领域，不含产品关联） ← 仅 In-Scope
Step 3   产品关联分析（按暂估分优先级搜索）           ← 仅 In-Scope
Step 3.5 问题域聚类与价值修正（仅触发的一级分类）     ← 新增
Step 2b  回填产品分 + 问题域修正，计算四维综合分
Step 4   "值得关注"清单提炼
Step 5   Excel 报告生成（9 sheets：Sheet 0 + Sheet 1~7 + Sheet 4.5）
Step 6   文字摘要与行动建议
```

### Step 0 — 输入解析与路径判断

1. **检测输入类型**：

   ```
   如果输入 Excel 包含"主标签"+"置信度"+"分类来源"列：
     → 快速路径 A：直接消费 classifier 输出
     → 从中提取：主标签、次要标签、置信度、同族组、领域范围
     → 跳过 Step 1，直接进入 Step 2

   否则：
     → 标准路径 B：原始专利清单
     → 进入 Step 1 调用 classifier
   ```

2. **补充字段识别**：无论哪种路径，扫描 monitor 特有的字段：

   | Monitor 特有字段 | 常见列名变体 |
   |-----------------|------------|
   | 同族数量 | 同族数量, Family Size, 同族数, family_size |
   | 布局国家 | 布局国家, Countries, 地域, geography |
   | 法律状态 | 法律状态, Legal Status, 状态 |

   （标题、摘要、申请人等通用字段的映射由 classifier 的 Step 0.1 处理）

3. **输入摘要**（非阻塞，输出后直接继续执行）：
   - 简要列出：专利总数、涉及的申请人、时间范围、缺失字段
   - 如果用户已在指令中指定重点关注领域 → 采用，否则默认全部 L0-L10
   - **不等待用户确认**，直接进入 Step 1 或 Step 2a
   - 仅当输入**缺少"标题"或"摘要"**时才阻塞等待用户补充数据

### Step 1 — 领域预筛 + 技术分类

**本步骤委托 patent-classifier 执行。**

调用 classifier，传入参数：
- 专利清单（原始输入）
- 标签体系：用户提供 或 默认 L0-L10
- `scope_filter = True`（激活领域相关性预筛）

Classifier 会执行其完整 Step 0-5 流程，输出丰富版结果，包含：

| Classifier 输出列 | Monitor 消费方式 |
|------------------|----------------|
| 主标签 | 技术领域归类，驱动 Sheet 3 分领域清单 |
| 次要标签 | 扩展产品关联搜索方向（Step 3） |
| 置信度 | 领域权重分修正（Step 2）+ 值得关注筛选（Step 4） |
| 同族组 | 产品关联搜索预算优化（Step 3） |
| 领域范围 | In-Scope / Out-of-Scope 过滤 |
| 分类来源 | 报告标注 |
| 标题/摘要/申请人/有效日期 | 元数据复用 |

**Monitor 在分类结果上的补充处理（两种路径通用）：**

无论是快速路径 A（直接消费 classifier 输出）还是标准路径 B（刚完成 classifier
调用），都执行以下检查：

1. **Out-of-Scope 摘要**（非阻塞）：
   - 输出 In-Scope / Out-of-Scope 分布摘要
   - 如 Out-of-Scope 占比 > 50%，在摘要中**标注提醒**（但不阻塞等待确认）
   - 用户可在后续任何时候要求重新纳入某些 Out-of-Scope 专利

2. **分类汇总**（非阻塞，输出后直接进入 Step 2a）：
   ```
   分类结果:
     纳入分析 (In-Scope): {N1} 件
       L1 核心部件: 8 件 (NVIDIA 5, Intel 3)
       L3 互连协议: 12 件 (...)
       Other-Computing: 3 件
     排除分析 (Out-of-Scope): {N2} 件
       自动驾驶感知: 12 件
       5G通信: 8 件
     分类置信度: 高 {X}件 / 中 {Y}件 / 低 {Z}件
   ```

后续 Step 2a-4 仅处理 In-Scope 专利（L0-L10 + Other-Computing）。

### Step 2a — 三维暂估评分

对每篇 In-Scope 专利先计算**不含产品关联**的三维暂估分，用于确定 Step 3
的搜索优先级。产品关联分在 Step 3 完成后由 Step 2b 回填。

#### 2.1 同族规模分 (Family Score)

反映专利权人对该发明的全球布局力度。

| 同族数量 | 得分 |
|---------|------|
| ≥ 10    | 100  |
| 7-9     | 80   |
| 4-6     | 60   |
| 2-3     | 40   |
| 1       | 20   |

若输入缺少同族数据，此维度跳过。

#### 2.2 布局国家分 (Geography Score)

五大关键法域各计 20 分：

| 法域 | 代表市场 | 单项分值 |
|------|---------|---------|
| US   | 北美    | 20      |
| CN   | 中国    | 20      |
| EP   | 欧洲    | 20      |
| JP   | 日本    | 20      |
| KR   | 韩国    | 20      |

布局 5 个法域 = 100 分，3 个 = 60 分，以此类推。包含 TW、IN 等其他法域可酌情加
5-10 分（上限 100）。

若输入缺少布局国家数据，此维度跳过。

#### 2.3 技术领域权重分 (Domain Relevance Score)

基于 classifier 输出的主标签判定，Out-of-Scope 专利不参与评分。

| 条件 | 得分 |
|------|------|
| 在用户指定的本期重点领域内 | 100 |
| 在 L0-L10 体系内但非本期重点 | 50 |
| 分类为 Other-Computing | 20 |

若用户未指定重点（使用全部 L0-L10），则落入任何 L0-L10 类别得 80 分，
Other-Computing 得 20 分。

**置信度修正**（利用 classifier 输出的置信度字段）：

```
如果置信度 = "高"：领域权重分 × 1.0（不修正）
如果置信度 = "中"：领域权重分 × 0.8
如果置信度 = "低"：领域权重分 × 0.5，且在报告中标注"分类待确认"
```

#### 2.4 暂估分计算与排序

```
暂估分 = Family × 0.40 + Geography × 0.33 + Domain(修正后) × 0.27
```

三维权重按原四维中去掉产品关联（25%）后等比放大。
当某维度数据缺失时，该维度权重按比例分配给其余维度。

**按暂估分降序排列**，此排序决定 Step 3 的搜索优先级。

### Step 3 — 产品关联分析

目标：识别每篇 In-Scope 专利可能关联的友商具体产品或技术。

**搜索策略：**

1. 从专利的标题、摘要中提取 2-3 个**核心技术关键词**（英文）
2. 结合友商产品列表（参考 `references/vendors.md`），构造搜索查询：
   ```
   {vendor} {keyword1} {keyword2} product
   ```
3. **次要标签扩展搜索**：如果 classifier 输出中有次要标签，额外搜索次要标签
   对应的产品方向。例：主标签 L1 GPU核心 + 次要标签 L10 安全 →
   同时搜索 GPU 产品和安全特性（如 Confidential Computing）
4. 基于搜索结果判定关联强度（强/中/弱）

**搜索预算控制：**

利用 classifier 输出的 `同族组` 字段优化搜索预算：

```
独立专利数 = 无同族组的专利 + 每个同族组计 1 件
总预算 = min(独立专利数 × 1.5, 40) 次 web_search
```

- 同一同族组内的专利共享产品关联结论——仅对每组的代表专利执行搜索
- 按 Step 2a **暂估分降序**安排搜索顺序，预算耗尽时低分专利标记为"弱关联"
- 对于技术方向明确且在 vendors.md 中有直接对应的（如 "NVLink" 明确是 NVIDIA
  互联产品），无需搜索即可判定强关联

**关联判定规则：**

- **强关联**：专利技术特征与某产品的已知规格/白皮书/技术文档直接匹配
  - 例：专利描述"多芯片间高速串行互联协议" + 申请人 NVIDIA → 强关联 NVLink
- **中关联**：技术方向一致但无法确认具体产品型号
  - 例：专利描述通用的"GPU 内存管理优化" + 申请人 AMD → 中关联 Instinct 系列
- **弱/无关联**：基础研究类专利，或技术距离当前产品线较远

### Step 3.5 — 问题域聚类与价值修正

**目的**：识别友商在某一技术领域的集中投入信号——当一批新专利
解决相似的技术问题，或单篇专利与特定产品/技术强绑定时，其战略价值
应高于孤立专利的简单加权分。

#### 3.5.1 触发条件

仅对**专利数量较多的一级分类**执行问题域聚类。默认阈值：

```
对每个一级分类（L0-L10），统计本期 In-Scope 专利数 N_L
  N_L ≥ 8  → 触发问题域聚类
  N_L < 8  → 跳过聚类，直接进入 Step 2b
```

阈值可由用户在调用时指定（如"对专利≥5件的领域都做聚类"）。
Other-Computing 类别也可单独触发，因为前沿探索类专利的小规模集中
往往更具战略意义——其默认阈值降低为 N ≥ 5。

#### 3.5.2 问题域提取与聚类

对触发的一级分类，执行三步聚类：

1. **技术问题提取**：对每篇专利，从摘要 + 第一权利要求（若有）中
   提取一句"技术问题描述"——即专利试图解决的核心问题。
   - 优先从摘要中"现有技术存在...问题"、"为了解决...问题"等句式提取
   - 若无显式问题陈述，从摘要的第一段技术目标推断
   - 输出格式："如何在 {场景} 下实现 {目标}，同时避免 {约束}"

2. **语义聚类**：将同一一级分类下所有专利的"技术问题"语义聚类。
   - 簇内距离判断标准：解决的核心技术问题是否相同（不要求技术手段相同）
   - 例：以下两篇属于同一问题域簇——
     - 专利 A："如何降低 GPU 间 AllReduce 通信的尾延迟"（硬件拓扑方案）
     - 专利 B："如何减少分布式训练中梯度同步的等待时间"（调度算法方案）
   - 簇大小要求：≥ 3 篇成簇，< 3 篇标记为"独立专利"

3. **簇命名与归档**：为每个问题域簇生成一句话名称，记录：
   - 簇名（如"L3-簇1: 高带宽互联尾延迟优化"）
   - 簇内专利列表
   - 簇内各申请人专利数（识别是单家集中还是多家共同关注）
   - 主要技术路径（硬件/协议/软件/算法）

#### 3.5.3 价值修正信号生成

基于聚类结果和 Step 3 的产品关联结果，为每篇专利生成**价值修正系数**：

| 修正信号 | 触发条件 | 系数 |
|---------|---------|------|
| **集中投入** | 专利属于一个 ≥ 3 篇的问题域簇，且簇内同申请人专利占 ≥ 60% | +10 |
| **行业聚焦** | 专利属于一个 ≥ 5 篇的问题域簇，且簇内涉及 ≥ 2 个申请人 | +8 |
| **强产品锚定** | Step 3 判定为强关联，且关联到具体产品型号（非系列）| +8 |
| **复合信号** | 同时满足"集中投入"或"行业聚焦"+"强产品锚定" | +15（覆盖前两者）|

修正系数累加上限：**+15**（避免单篇专利通过多重信号过度加分）。

不满足任何信号的专利修正系数为 0，价值得分不变。

### Step 2b — 回填产品分，计算四维综合分 + 问题域修正

Step 3 和 Step 3.5 完成后，回填产品关联分并计算最终综合分。

#### 产品关联分 (Product Association Score)

| 关联强度 | 得分 | 判定标准 |
|---------|------|---------|
| 强关联  | 100  | 专利技术与已知产品特性直接对应 |
| 中关联  | 60   | 技术方向一致但未确认具体产品 |
| 弱/无关联 | 20 | 未找到明确的产品关联证据 |

#### 四维综合分计算（含问题域修正）

```
基础分 = Family × 0.30 + Geography × 0.25 + Domain(修正后) × 0.20 + Product × 0.25
综合分 = min(基础分 + 问题域修正系数, 100)
```

当某维度数据缺失时，该维度权重按比例分配给其余维度。
问题域修正系数来自 Step 3.5（范围 0~+15），未触发聚类的专利修正为 0。

#### 重要性等级

| 等级 | 分数区间 | 含义 |
|------|---------|------|
| ★★★ 高度关注 | ≥ 75 | 多维度均表现突出，建议立即深入分析 |
| ★★ 中度关注  | 50-74 | 部分维度突出，建议纳入跟踪清单 |
| ★ 一般关注   | < 50  | 暂无突出表现，常规归档 |

### Step 4 — "值得关注"清单提炼

从以下两个来源筛选值得关注的专利：

**来源 A — Other-Computing 专利**（原有逻辑）

从分类为 "Other-Computing" 的专利中，满足任一条件即纳入：
1. 产品关联分 ≥ 60
2. 同族规模分 ≥ 80
3. 布局包含 3+ 关键法域
4. 涉及新兴技术方向（量子计算、光互连、存算一体、chiplet 等）

**来源 B — 低置信度 In-Scope 专利**（新增）

Classifier 输出置信度为"低"的 In-Scope 专利，说明其分类结果不确定，可能被
错误归类。满足以下条件时纳入值得关注清单：
- 置信度 = "低" 且 综合分 ≥ 50（即其他维度表现尚可）
- 在关注理由中注明"分类置信度低，建议人工复核技术领域归属"

对筛选出的专利，额外撰写一段简短的"关注理由"（2-3 句）。

### Step 5 — Excel 报告生成

生成一份多 Sheet 的 `.xlsx` 文件，包含以下工作表：

#### Sheet 0: 五类启示框架 (Five Insights)

这是报告的**入口导航页**，位于所有 Sheet 最前面，让读者第一眼就能看到
本期分析的核心结论和行动指引，再按需跳转到各 Sheet 查证。

结构为五个横向展开的启示区块，每个区块包含：

| 字段 | 内容 |
|------|------|
| 启示编号 | ① ② ③ ④ ⑤ |
| 启示类型 | 布局热度快照 / 高价值早期预警 / 友商产品路线反推 / 技术路径差异化 / 非显性信号 |
| 核心问题 | 本启示回答的业务问题（一句话） |
| 一句话结论 | 本期数据驱动的具体结论（由 Step 6 回填） |
| 关键数据依据 | 支撑结论的 2-3 个关键数字或专利号 |
| 数据来源 Sheet | 可跳转的具体 Sheet 名称和位置 |
| 行动建议 | 立即 / 短期 / 战略 三类行动项各一条 |

### Step 6 — 文字摘要与行动建议

在对话中输出一段结构化的文字摘要，**严格按五类启示框架组织**，
确保每类启示都有明确的结论、依据和行动建议。

**6.0 概览**（背景信息，非启示）
- 本期分析范围（时间、公司、专利数）
- 范围过滤结果：总输入 N 篇，纳入分析 N1 篇，Out-of-Scope N2 篇
- 分类质量：置信度分布（高/中/低）

**启示 ① 布局热度快照**

回答："本期友商在哪些技术领域投入最多？谁最活跃？"

输出要点：
- Top 3 最热一级分类，按 In-Scope 专利数降序，说明主要申请人构成
- 若某领域出现明显数量激增，特别标注为"异常热点"
- 一句话结论 + 关键数据 + 行动建议回填到 **Sheet 0 启示①区块**

**启示 ② 单篇高价值专利的早期预警**

回答："哪些单篇专利最值得立即跟进？"

输出要点：
- 综合分 Top 5 的 ★★★ 专利，每篇 1-2 句话说明价值来源
- 对每篇明确建议下一步动作：CFP 侵权评估 / 无效检索预案 / 持续跟踪
- 标注问题域修正系数 ≥ +10 的专利
- 标注低置信度进入 Top 的专利为"分类待确认，建议人工复核"
- 一句话结论 + 关键数据 + 行动建议回填到 **Sheet 0 启示②区块**

**启示 ③ 友商产品路线的反推**

回答："友商下一代产品/技术走向是什么？"

输出要点：
- 按友商归纳：每家公司本期专利集中保护的产品线/技术方向
- 重点列出强关联且关联到具体产品型号（非系列）的专利组
- 识别"产品线动作密集"信号：某友商在某产品线 ≥ 3 件强关联专利
- 次要标签揭示的跨域布局趋势
- 一句话结论 + 关键数据 + 行动建议回填到 **Sheet 0 启示③区块**

**启示 ④ 技术路径的差异化判断**（仅当 Step 3.5 触发时输出）

回答："同一个技术问题，各家用了哪些不同手段？谁的方案更先进？"

输出要点：
- 列出本期所有问题域簇，按簇内专利数降序
- 每个簇说明：簇名、所属一级分类、专利数、申请人分布、技术路径分布
- 区分**单家集中型**（友商即将发布相关产品的强信号）和**行业聚焦型**
  （行业级技术拐点信号）
- 对单家集中型提示"我方是否有储备"；对行业聚焦型提示"路径空白布局机会"
- 一句话结论 + 关键数据 + 行动建议回填到 **Sheet 0 启示④区块**
- 未触发时 Sheet 0 启示④区块显示"本期未触发问题域聚类（各领域专利数均 < 8）"

**启示 ⑤ 非显性信号的捕捉**

回答："哪些容易被忽略但战略价值高的专利需要关注？"

输出要点：
- 来源 A：Other-Computing 前沿探索专利，重点强调量子/光互连/存算一体等新兴方向
- 来源 B：低置信度高分专利（综合分 ≥ 50），提示"可能是占坑式申请或跨领域核心技术"
- 来源 C：Out-of-Scope 异常集中（若有），作为战略情报提及
- 一句话结论 + 关键数据 + 行动建议回填到 **Sheet 0 启示⑤区块**

## 搜索预算全局规划

```
阶段 1 — Classifier 分类（仅标准路径 B 需要）:
  预算: min(需检索专利数 × 2, 30) 次
  如果用户数据已有标题+摘要 → 此阶段消耗 ≈ 0

阶段 2 — Monitor 产品关联:
  预算: min(独立专利数 × 1.5, 40) 次

总预算上限: ≤ 70 次
```

注：Step 3.5 问题域聚类为**纯语义分析**，不消耗任何 web search 预算。

## 实践经验与注意事项

1. **快速路径优先**。如果用户已经用 classifier 分类过，直接用其输出作为
   monitor 输入，可跳过分类阶段，节省大量 token 和搜索预算。

2. **两阶段评分打破循环依赖**。Step 2a 用三维暂估分（不含产品关联）为 Step 3
   的搜索排优先级；Step 3 完成后 Step 2b 回填产品分计算最终四维综合分。
   不要试图在 Step 2 一次算完——产品关联分此时尚不存在。

3. **非阻塞式执行**。Step 0 的输入摘要和 Step 1 的分类汇总均为非阻塞——输出后
   直接继续执行，不等待用户确认。仅在缺少标题/摘要等关键数据时才阻塞。

4. **问题域聚类是免费的洞察**。Step 3.5 不消耗 web search 预算，仅做语义聚类，
   但能识别两类高价值信号——单家集中投入（产品发布预警）和行业聚焦
   （技术拐点信号）。仅对 ≥8 件的一级分类触发，避免小样本聚类的噪音。

5. **修正系数有上限**。问题域修正最多 +15 分，让"集中布局 + 产品锚定"的专利
   能突破单纯加权分的局限，但不至于颠覆基础评分。

6. **同族组是搜索预算的倍增器**。Classifier 输出的同族组字段让 monitor
   的产品关联搜索可以按组复用，通常节省 20-30% 的搜索次数。

7. **vendors.md 是产品关联的第一参考**。搜索前先检查是否能直接匹配，减少
   不必要的 web search。产品信息必须基于可查询公开资料核实，不得虚构。

8. **"Other-Computing" 不等于不重要**。有些前沿探索性专利（如光互连、量子纠错）
   可能不在现有 L0-L10 体系中，但恰恰代表友商的前瞻布局，Step 4 的筛选确保
   这些不被遗漏。

9. **低置信度是信号不是错误**。Classifier 标记为低置信度的专利不代表分类
   一定错误，而是提醒人工关注。Monitor 通过置信度修正评分 + 值得关注清单
   双重机制处理这种不确定性。

10. **同族数量是最可靠的重要性信号之一**。一个公司愿意在 5+ 法域为同一发明申请
    专利，通常意味着该技术有重大商业价值。

11. **中国专利需注意中英文术语桥接**。如果输入含中文专利，在产品关联搜索时需要
    将中文技术术语翻译为英文后再搜索友商产品文档。

12. **报告生成使用 openpyxl**。读取 xlsx 用 openpyxl，确保安装：
    `pip install openpyxl --break-system-packages`

13. **Sheet 0 最后写入**。五类启示框架（Sheet 0）在 Step 6 文字摘要生成后写入，
    确保 Excel 中的结论与对话摘要完全一致。Sheet 0 是用户打开报告的第一眼，
    内容质量直接决定报告可读性。
