# 专利Idea胶片分析 Skill

## 概述

本Skill对专利Idea胶片进行全面预审，输出两大块分析：

1. **胶片撰写质量检查**：6项逐一审核，发现撰写问题
2. **创新性深度分析**：三步法提炼发明点，检索对比文件，评估创造性

适用技术领域：存储（SSD、HDD、分布式存储、存储协议等）、AI（推理优化、AI存储、模型训练等），以及其他技术领域。

---

## 第一步：文件提取

> ⚠️ **文件名编码注意**：胶片文件名可能含中文、空格或括号。操作文件前先用
> `os.listdir()` 列出目录并从结果中**复制**文件名，不要手动拼接；
> 传路径给 python-pptx / python-docx 时使用 `pathlib.Path` 对象而非字符串，
> 可避免部分环境下的编码转换错误。

根据文件类型调用对应技能提取胶片全文：

- **PPT格式**：调用 pptx 技能提取内容，使用 `python -m markitdown presentation.pptx`
- **Word格式**：调用 docx 技能提取内容，使用 `pandoc` 或 unpack for raw XML

提取后通读全文，理解胶片整体结构和技术方案。

---
## 第二步：创新性分析（三步法 + 检索 + 对比分析）

> ⚠️ 执行检索前，先确认 `exa_web_search_exa` / `exa_web_fetch_exa` 工具可用。若不可用，在报告开头明确告知用户，跳过检索环节，仅基于已有知识完成其余分析。
>
> ⚠️ 必须在质量检查之前完成，因为发明点提炼的结果会直接影响质量检查中保护点的评判。
>
> ⚠️ 若胶片内容过少（不足5页或关键信息缺失），三步法可能无法充分提炼delta。此时应在报告中明确指出"胶片内容不足以完成完整分析"，并列出缺失的关键信息，供用户补充。

### 一、三步法提炼发明点（必做，不得跳过）

> ⚠️ 这是整个分析的基础。delta提炼不准，后续所有分析都会偏。

**Step 1 — 读背景技术**：确定方案所属技术领域，理解现有技术做到了什么、存在什么问题或局限性。

**Step 2 — 读技术方案**：理解本发明的方案是什么。

**Step 3 — 读有益效果**：理解本发明声称带来了哪些改进（性能、可靠性、成本、功耗、精度等）。

**Step 4 — 计算delta**：只有"先前技术做不到、本发明做到了"的部分才构成真正发明点。常规实现细节、标准协议规定的行为、业界公知的优化手段均不算发明点。

输出格式：
```
【发明点提炼】
- 背景技术的局限：...
- 发明概述（100~200字）：在...场景中，一般通过...方案解决...问题，传统方案因...导致...，本方案采用了...，解决了...。
- 本专利方案（300字）
  - 创新点：通过...技术手段，实现...
  - 核心处理流程：步骤1 → 步骤2 → 步骤3（链条式说明）
  - 关键改进：对比现有技术的3~5个核心差异点，每个差异点用3~5句话说明
- 真正的发明点（delta）：...
```

### 二、检索策略

#### 2.1 检索范围

对比文件不限于专利，**所有公开发表的资料**均可作为对比文件，包括：

| 资料类型 | 典型来源 |
|---------|---------|
| 专利文献 | Google Patents、USPTO、EPO、国家知识产权局 |
| 学术论文 | arxiv、IEEE Xplore、ACM Digital Library |
| 顶会论文 | 各技术领域顶会（如存储：FAST/USENIX ATC/ISCA/OSDI/SC；AI：NeurIPS/ICML/ICLR；网络：SIGCOMM/NSDI 等） |
| 技术标准 | NVMe Spec、JEDEC标准、SNIA白皮书 |
| 厂商公开资料 | 技术白皮书、产品手册、官方技术博客 |
| 开源项目文档 | GitHub技术文档、设计文档（RFC等） |

#### 2.2 搜索执行原则

> ⚠️ **本模块所有检索必须使用 EXA MCP 工具执行**，禁止使用其他搜索工具。
> 若 EXA MCP 工具不可用，必须在报告开头明确告知用户，不得静默回退。

| 工具名 | 用途 |
|--------|------|
| `exa_web_search_exa` | 通用网络搜索，返回摘要+URL |
| `exa_web_fetch_exa` | 抓取指定 URL 的完整页面内容（用于全文阅读） |

**搜索角度一：技术问题角度**（第1轮搜索）：
- 用"技术领域 + 解决的技术问题"作为关键词，**每轮至少换2~3组不同关键词**
- 目标：相同技术领域下，找到解决同一问题的现有技术
- 每轮调用：`exa_web_search_exa(query="...", numResults=8)`，此角度≤5次

**搜索角度二：技术方案角度**（第2轮搜索）：
- 用"核心技术机制+实现手段"作为关键词，每轮至少换2~3组不同关键词
- 目标：找到采用类似技术手段的现有方案，不限于专利
- 每轮调用：`exa_web_search_exa(query="...", numResults=8)`，此角度≤5次

**补充搜索**（第3~4轮，按需执行）：
- 前两轮不足5篇时，换关键词或换资料类型再搜
- 针对某个具体区别特征单独搜索，寻找D2候选
- 或针对IPC分类号定向搜索专利库

**全文抓取（必做，不可跳过）**：
- 对候选中最相关的2~3篇（尤其D1和D2），**必须用 `exa_web_fetch_exa` 抓取全文**
- 禁止仅凭搜索摘要/高亮判定相关度——摘要可能遗漏关键区别
- Google Patents 示例：`exa_web_fetch_exa(url="https://patents.google.com/patent/US11194679B2/en")`

**搜索总控**：`exa_web_search_exa` 共5~10次 + `exa_web_fetch_exa` 2~3次，确保3~5篇有实质内容的对比文件后停止。

#### 2.3 IPC分类号参考（专利检索时使用）

| 技术方向 | IPC分类 |
|---------|---------|
| 存储控制/管理 | G06F 3/06 |
| Flash存储 | G11C 16/00 |
| 纠错/数据保护 | G11C 29/00 |
| 数据压缩/重删 | G06F 12/00 |
| 存储网络协议 | H04L 29/00 |
| AI/机器学习 | G06N 3/00 |
| 分布式存储 | G06F 15/16 |

若方案不属于存储领域，应通过 `exa_web_search_exa` 先查找该领域的IPC分类号再定向检索。

### 三、对比文件分析

#### 3.1 相关度判断标准

| 相关度 | 判断依据 |
|--------|---------|
| **高** | 同一技术领域，解决相同或高度相似的技术问题，技术手段有实质重叠 |
| **中** | 同一技术领域但解决不同问题，或不同领域但采用相似技术手段 |
| **低** | 仅部分术语或概念相关，技术思路本质上不同 |

> ⚠️ **相关度判断必须在 `exa_web_fetch_exa` 抓取全文确认后才能最终确定**。尤其D1必须抓全文确认。

#### 3.2 对比文件详析格式

```
【对比文件 D1/D2/...】
- 来源：专利号 / 论文标题 / 申请人 / 公开日
- 技术方案摘要：（2-3句，说清楚它做了什么）
- 与本发明的相同点：...
- 与本发明的区别：...
- 相关度：高 / 中 / 低（依据全文确认后的判断）
```

### 四、创造性分析

#### 4.1 区别技术特征

以D1为基础，逐条列出本发明相对于D1的区别技术特征：

```
区别特征1：[技术特征描述，用技术方案语言，不用效果语言]
区别特征2：...
```

#### 4.2 技术效果对应

每条区别特征对应具体可测量的技术效果：

```
区别特征1 → 技术效果：...（定性表述，避免出现具体数字百分比）
```

#### 4.3 显而易见性判断

| 维度 | 评估问题 |
|------|---------|
| 技术启示 | D1或D2是否提供了明确的技术启示？ |
| 常规组合 | 是否只是将现有技术的简单叠加？ |
| 技术偏见 | 是否克服了领域内公认的技术偏见或反直觉的选择？ |
| 意外效果 | 是否产生了预料不到的技术效果？ |

#### 4.4 创造性综合评分

```
【创造性评估结论】
综合评分：★★★★☆（1-5星）

- 创造性强项：...（难以被D1+D2组合覆盖的核心特征）
- 创造性风险：...（容易被审查员挑战的薄弱点）
- 建议保护策略：独立权利要求应聚焦于...
```

### 五、保护点重写建议

#### 5.1 诊断原有保护点

| 原有写法类型 | 典型问题 | 示例 |
|------------|---------|------|
| 效果型 | 写的是达到了什么，不是做了什么 | "提高了系统效率" |
| 功能型 | 写的是能做什么，不是怎么做的 | "能够自动识别关键数据" |
| 场景限定 | 不必要地限窄了保护范围 | "在特定硬件介质中…" |
| 概念型 | 没有具体技术特征支撑 | "采用智能调度机制" |

#### 5.2 重写建议

基于delta，给出保护点的重写建议：
- 描述**技术特征**：系统/装置的结构特征，或方法的步骤特征
- 去掉不必要的场景限定
- 独立保护点聚焦最核心的delta
- 从属保护点方向列举2-3个附加特征方向

### 六、实施例扩展方向

#### 6.1 创新点与技术效果

| 创新点 | 对应的技术效果 |
|--------|--------------|
| [技术特征描述] | [有益效果，定性表述，避免数字] |

> 注意：有益效果避免出现具体数字（如"准确率提升X%"），用定性表述。

#### 6.2 发散思维

基于核心方案进行前瞻性发散，选择与本方案最相关的3~5个维度，每个1~2段。

参考方向：技术领域扩展、系统架构扩展、实现方式变体、算法/数据结构变体、操作模式扩展、
性能优化变体、协议与接口扩展、全流程扩展、应用场景扩展、安全与可靠性扩展。

#### 6.3 发散输出格式

```
**发散点1：[方向名称]**
[1-2段描述]
```

#### 6.4 附加技术特征（从属权利要求方向）
- 附加点1：[具体技术特征]（技术效果：...）
- 附加点2：[具体技术特征]（技术效果：...）

#### 6.5 延伸发明方向（系列专利布局）
- 延伸方向1：[独立发明描述]，可解决的问题：...
- 延伸方向2：[独立发明描述]，可解决的问题：...

## 第三步：胶片质量检查（6项标准）

每项给出判定：**满足 / 部分满足 / 不满足**，并标注具体问题。
违反重要注意事项的项目标注【⚠️ 注意】。
无论保护点检查结果如何，报告第五章必须给出**保护点重写建议**。

### 检查项1：Idea合规检查页

**检查内容**：
- 胶片中是否存在合规检查页（checklist页面）
- 检查项是否包含：是否引用外来非公开信息、职务成果归属、是否涉及出口管制、发明人信息是否准确

**判定标准**：
- 满足：合规页存在，检查项完整
- 部分满足：合规页存在但检查项不完整
- 不满足：无合规页

### 检查项2：背景与现有技术

**检查内容**：
- 是否描述了问题产生的根源
- 现有技术的问题是否为**客观技术问题**（时延高、成本高、可靠性低、资源浪费等），而非主观感受【⚠️ 注意】
- 现有技术存在的问题是否正好是本发明能解决的问题【⚠️ 注意】
- 是否介绍了**业界同类技术**（不只是自家方案，还包括竞争对手或业界主流方案）

**判定标准**：
- 满足：4点均覆盖，描述客观准确
- 部分满足：存在主观描述、遗漏业界技术、或问题与方案不匹配
- 不满足：背景技术缺失或严重偏差

### 检查项3：技术方案

**检查内容**：
- 方案是否完整可实现，不能只是概念或仅揭露问题【⚠️ 注意】
- 是否图文结合，不能只有文字或只有图
- 本领域普通技术人员能否看懂并复现
- **方案边界是否清晰**：有没有把现有技术误包进来当成自己的创新【⚠️ 注意】

**判定标准**：
- 满足：方案完整、图文结合、边界清晰
- 部分满足：方案基本完整但图文不配合，或边界模糊
- 不满足：方案不完整、仅有概念、或无法复现

### 检查项4：技术保护点

**检查内容**：
- 保护点写的是**技术特征**还是优点/功能/效果【⚠️ 注意】
- 是否层次化撰写，有独立保护点和从属保护点的区分
- **有没有把实施场景写进保护点**限窄范围【⚠️ 注意】

**判定标准**：
- 满足：保护点为技术特征、层次化、无不必要场景限定
- 部分满足：存在部分效果/功能描述，或层次化不足
- 不满足：保护点全部写的是效果、优点或功能

### 检查项5：技术效果与市场价值

**检查内容**：
- 问题→方案→效果→市场价值逻辑链是否完整自洽
- 技术效果描述是否具体，**有没有可比较的基准**【⚠️ 注意】
- 市场价值是否有量化分析或明确的应用场景

**判定标准**：
- 满足：逻辑链完整、效果有基准、市场价值清晰
- 部分满足：逻辑链存在断点，或效果无基准，或市场价值模糊
- 不满足：效果与方案脱节，或无市场价值分析

### 检查项6：取证方法

**检查内容**：
- 是否针对每个保护点都给出了对应的取证方式【⚠️ 注意】
- 取证方式是否具体可操作（测试步骤、抓log方法、拆机分析路径等）
- **取证难度评估**：
  - 硬件/系统领域：拆机、抓log、测试、产品手册、性能测试是否真实可行
  - 软件/算法领域：黑盒情况下是否真的能取证；若无法取证需标注风险【⚠️ 注意】

**判定标准**：
- 满足：每个保护点均有具体取证方法，难度评估合理
- 部分满足：取证方法不完整，或存在黑盒无法取证的保护点未被识别
- 不满足：无取证方法，或取证方法完全不可操作

## 第四步：输出报告

### 4.1 生成Markdown报告文件

报告包含以下8章：
1. **封面信息**：专利名称、核心技术方向、分析日期
2. **发明点提炼**：背景局限、发明概述、本方案、delta表格
3. **胶片撰写质量评价**：6项检查表格，每项判定+评语
4. **现有技术对比分析**：对比文件列表 + 逐篇详析 + 区别特征 + 显而易见性判断
5. **创造性综合评估**：综合评分（1-5星）+ 强项 + 风险点 + 保护策略
6. **保护点重写建议**：问题诊断 + 重写示范
7. **实施例扩展方向**：发散思维 + 从属方向 + 延伸布局
8. **总结与优先改进事项**：总体评价 + 高/中/低优先级事项

**文件命名**：`{发明核心内容}-预审报告.md`

> ⚠️ Markdown报告内容较长，必须写入文件而非仅打印到对话中。
> 用 `write` 工具写入 `.md` 文件；若过长触发JSON限制，可分段写入或使用 `task` 工具生成。
> 保存路径：`{工作目录}/{发明核心内容}-预审报告.md`

### 4.2 询问用户是否需要Word版本

Markdown报告保存后，告知用户路径并询问："需要将报告转为Word格式（.docx）吗？"
- 不需要：结束流程
- 需要：执行4.3转换

### 4.3 转换Markdown为Word（按需）

使用 Python `python-docx` 库生成Word文档。若未安装，先执行 `python -m pip install python-docx`。

转换要点：
- 编写 `.py` 脚本保存到临时路径，用完整Python路径执行
- 用 `python-docx` 设置样式（标题字体、表格、颜色编码等）
- 判定颜色：满足→#2E7D32，部分满足→#E65100，不满足→#C62828
- 页眉：专利Idea胶片预审报告
- 页脚：页码
- 文件命名：`{发明核心内容}-预审报告.docx`
- 页面：A4，1英寸页边距

```python
from docx import Document
from docx.shared import Pt, Cm, RGBColor
from docx.enum.text import WD_ALIGN_PARAGRAPH
from docx.oxml.ns import qn

doc = Document()
s = doc.sections[0]
s.page_width = Cm(21)
```
